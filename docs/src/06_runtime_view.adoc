=== User Registration and Login
Both flows follow the same pattern: the client first fetches a CSRF token, then submits credentials. The Gateway validates the CSRF token, forwards the request to the Auth Engine, and on success creates a server-side session in Redis, returning an httpOnly session cookie. The only difference is what the Auth Engine does with Firebase.

[plantuml, target=runtime_auth, format=png]
....
@startuml
autonumber
skinparam shadowing false
skinparam sequenceMessageAlign center
skinparam ParticipantBackgroundColor #F5F5F5
skinparam ParticipantBorderColor #424242
skinparam DatabaseBackgroundColor #EEEEEE
skinparam DatabaseBorderColor #424242
skinparam SequenceArrowColor #424242

actor "Player" as Client
participant "Users\n(Gateway)" as Gateway
participant "Auth Engine" as Auth
database "Firebase" as Firebase
database "Redis" as Redis

Client -> Gateway : GET /api/csrf-token
Gateway --> Client : csrf_token cookie + csrfToken body

alt Registration
    Client -> Gateway : POST /api/register\nHeader: X-CSRF-Token
else Login
    Client -> Gateway : POST /api/login\nHeader: X-CSRF-Token
end

Gateway -> Gateway : verifyCsrf()\n(cookie csrf_token == header X-CSRF-Token)

alt Registration
    Gateway -> Auth : POST /register
    Auth -> Firebase : Store new user credentials
    Firebase --> Auth : 201 Created + user profile
else Login
    Gateway -> Auth : POST /login
    Auth -> Firebase : Verify credentials
    Firebase --> Auth : 200 OK + user profile
end

Auth --> Gateway : username + email
Gateway -> Redis : SETEX session:{id} → {username, email} (TTL 30 min)
Gateway --> Client : 200 OK + sessionId cookie (httpOnly, Secure, SameSite=Lax)
@enduml
....

=== Post-Login Session Usage
After login, every request that reads or mutates user state goes through the session stored in Redis. Read requests only need the `sessionId` cookie; state-changing requests additionally require the `X-CSRF-Token` header to match the `csrf_token` cookie.

[plantuml, target=runtime_session_lifecycle, format=png]
....
@startuml
autonumber
skinparam shadowing false
skinparam sequenceMessageAlign center
skinparam ParticipantBackgroundColor #F5F5F5
skinparam ParticipantBorderColor #424242
skinparam DatabaseBackgroundColor #EEEEEE
skinparam DatabaseBorderColor #424242
skinparam SequenceArrowColor #424242
skinparam ActorBorderColor #424242

actor "Player" as Client
participant "Users\n(Gateway)" as Gateway
database "Redis" as Redis

note over Client, Redis: 1 — Read current session (e.g. on page load after login)
Client -> Gateway : GET /api/me\nCookie: sessionId
Gateway -> Redis : GET session:{sessionId}
Redis --> Gateway : { username, email }
Gateway --> Client : 200 OK { username, email }

note over Client, Redis: 2 — State-changing request (CSRF + session both required)
Client -> Gateway : POST /api/update-username\nCookie: sessionId + csrf_token\nHeader: X-CSRF-Token
Gateway -> Gateway : verifyCsrf()\n(cookie csrf_token == header value)
Gateway -> Redis : GET session:{sessionId}
Redis --> Gateway : { username, email }
Gateway -> Redis : SETEX session:{sessionId} (TTL refreshed, username updated)
Gateway --> Client : 200 OK { username }

note over Client, Redis: 3 — Logout
Client -> Gateway : POST /api/logout\nCookie: sessionId + csrf_token\nHeader: X-CSRF-Token
Gateway -> Gateway : verifyCsrf()
Gateway -> Redis : DEL session:{sessionId}
Gateway --> Client : 200 OK\nSet-Cookie: sessionId (cleared)
@enduml
....

=== Match Creation
This view illustrates how the system initializes a game session. The Users Gateway routes the request to the Game Manager, which prepares the environment for a new match.

Resource Allocation: The Game Manager creates a unique match ID.

Fast Persistence: The initial state is stored in Redis to allow for high-frequency updates during gameplay.

=== Player Move
The player move scenario represents the core interactive loop of the application. It involves real-time validation and state synchronization across the engine and the cache.

State Retrieval: The Game Manager pulls the current board status from Redis.

Engine Validation: The Gamey Engine processes the move logic to ensure it follows game rules.

Cache Update: The new state is pushed back to Redis for immediate availability.

=== Bot Move
This scenario covers automated gameplay. It is typically triggered internally by the Game Manager once a player's turn is completed.

Decision Making: The Bot Logic within the Engine calculates the optimal response.

Asynchronous Sync: The resulting state is synchronized with the Redis cache so the player sees the bot's reaction upon their next refresh or through a socket.

=== Game End and Result Persistence
This scenario describes what happens when the final move of a match is played. The Gamey Engine detects the win condition, and the Game Manager persists the result to Firebase Firestore before clearing the now-finished match from Redis.

participant "Game Manager" as GM
participant "Gamey\n(Engine)" as Engine

database "Firebase\n(Firestore)" as Firebase

Client -> Gateway : POST /game/move (winning move)
Gateway -> GM : Proxy: /move
GM -> Redis : Get Current State
GM -> Engine : Validate & Execute Move
Engine --> GM : New State + WIN detected
GM -> Firebase : Persist match result\n(winner, duration, moves, difficulty)
Firebase --> GM : Saved
GM -> Redis : DEL game:{matchId} (release active state)
GM --> Gateway : 200 OK (final result)
Gateway --> Client : Final Board State + Winner
@enduml
